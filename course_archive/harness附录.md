# 附录 A：输入文件规范

## A.1 真值表文件
- 格式：Markdown 表格，位于"真值表与逻辑表达式"文件夹内
- Agent 在运行程序前需先将 md 表格解析为 CSV（存放在对应"实验x"文件夹内，命名 `truth_table.csv`），作为程序2 校验的标准
- CSV 第一行为表头，列顺序：所有输入变量 → 所有输出变量
- 变量名以表达式文件中的标注为准

## A.2 逻辑表达式文件
- 表达式已按指定器件类型写好（如 7400 对应与非式，7420 对应四输入与非式），可直接用于门级映射，无需再做代数变换
- 文件中会明确标注：
  - 使用的器件型号与数量（例：`7400 × 2, 7420 × 1`）
  - 输入变量列表（例：`inputs: A, B, C`）
  - 输出变量列表（例：`outputs: F`）
  - 每个输出对应的表达式

## A.3 spec.json（可选，由 agent 生成）
- Agent 可根据你口头或文件中的自然语言描述，自动在"实验x"文件夹内生成 `spec.json`，字段包含：真值表路径、表达式文件路径、器件清单、输入/输出变量、期望迭代上限（默认 3）
- 你不需要手写 spec.json，它仅作为程序1、程序2、agent 三方共享的结构化记录

# 附录 B：中间表示与输出格式

## B.1 芯片实例命名
- 统一采用 `型号_字母` 格式，如 `7400_A`、`7400_B`、`7420_A`
- 一旦实例确定，不得增删、不得改名，agent 仅能修改连线

## B.2 电路连接输出（程序1 必须同时产出两份）
1. **人类可读文本**：`circuit.txt`，每行一条连接，格式
   ```
   7400_A.3 -> 7420_A.1
   INPUT.A  -> 7400_A.1
   7420_A.8 -> OUTPUT.F
   ```
2. **结构化 JSON 网表**：`netlist.json`，结构示例
   ```json
   {
     "chips": [{"name":"7400_A","type":"7400"}, ...],
     "inputs": ["A","B","C"],
     "outputs": ["F"],
     "connections": [
       {"from":{"chip":"7400_A","pin":3}, "to":{"chip":"7420_A","pin":1}},
       ...
     ]
   }
   ```
- 两份文件一一对应，程序2 以 `netlist.json` 作为输入

## B.3 电源 / 地 / NC 引脚
- 仿真时忽略，但 `netlist.json` 与 `circuit.txt` 中必须显式记录（如 `7400_A.14 -> VCC`、`7400_A.7 -> GND`、`7420_A.3 -> NC`）
- 便于后续工艺校验与日志完整性

## B.4 连接输出排序规则
- `circuit.txt` 与日志中的"完整电路连接方式"均按下列顺序排列：
  1. 先按芯片实例名字典序（`7400_A` < `7400_B` < `7420_A`）
  2. 同一芯片内按引脚编号升序（1,2,3,…,14）
- 这是日志中"按顺序"的明确定义

# 附录 C：程序职责划分

## C.1 程序1（综合器）
- 输入：逻辑表达式文件 + 器件清单
- 处理：表达式 → AST → 按器件类型直接映射到门 → 引脚分配
- 输出：`circuit.txt` + `netlist.json`
- 独立运行，不读真值表

## C.2 程序2（验证器）
- 输入：`netlist.json` + `truth_table.csv`
- 处理：解析网表 → 拓扑排序 → 对每组输入组合求值 → 生成实际真值表 → 与期望真值表 diff
- 输出：`verify_report.json`（含 diff 行号、错误位置）+ `actual_truth_table.csv`
- 必须能脱离程序1 独立运行（仅依赖网表文件）

## C.3 器件库扩展
- 器件定义放在 `模拟电路项目/chips/` 下，每个型号一个文件（如 `chip_7400.py`、`chip_7420.py`）
- 定义内容：引脚规格（INPUT/OUTPUT/VCC/GND/NC）、门的逻辑函数
- 新增器件只需新增文件并在注册表登记，不改动程序1、程序2 核心逻辑

# 附录 D：Agent 修正流程

## D.1 修改权限
Agent 可执行的操作（优先级从低到高）：
1. 修改 `netlist.json` 中的连线
2. 回到程序1 重新生成整个网表
3. 复制原始代码至"实验x"文件夹并修改副本

芯片实例（名字、型号、数量）**不可更改**。

## D.2 迭代状态机
- 每一轮状态：`{round_id, netlist, expected_tt, actual_tt, diff, hypothesis, action}`
- 最大迭代轮次：**3 轮**
- 若 3 轮后仍未通过，终止并在日志中写明"无法自动修正"及原因分析

## D.3 每轮工作流
1. 运行程序1 → 产出网表
2. 运行程序2 → 产出实际真值表与 diff
3. 若 diff 为空：进入成功流程（见 D.5）
4. 若 diff 非空：定位错误门/错误连线 → 提出修改假设 → 写本轮日志 → 执行修改 → 下一轮

## D.4 代码更新建议
- 若 agent 判断问题源于原始代码缺陷（而非本实验连线问题），在日志中：
  - 给出修改后的完整代码
  - 给出正确性论证（为什么这样改能修复，且不破坏其他情况）
- **必须等待用户同意后才能回迁到 `模拟电路项目/` 下的原始代码**
- 未获同意前，修改仅存在于"实验x"文件夹内的副本

## D.5 成功流程
- 生成 `实验x_日志_成功.md`，包含：
  - 最终表达式与器件清单
  - 完整电路连接方式（按 B.4 顺序）
  - 真值表验证通过证明
  - 若涉及代码改动：附代码 diff 与正确性论证

# 附录 H：扩展能力（Step 2-4 增量，对 A.2 / B.3 / C.3 的补充）

> 本附录仅追加新条目，不改动 A.1–G 的任何已有文字。所有"扩展"是增量项，已实现并已通过回归。

## H.1 抽象层增量

在原 `Gate`（单输出小规模逻辑门）旁，新增一个平行抽象 `Block`：

- `Gate`：1 输出，例如 7400 的 NAND2、7420 的 NAND4
- `Block`：N 输入 → M 输出，承载译码器、MUX 等中规模器件，例如 74138 的 DECODE3、74153 的 MUX4

`ChipSpec` 同时持有 `gates: List[Gate]` 与 `blocks: List[Block]`。7400 / 7420 的 `blocks` 为空，行为完全不变（向后兼容）。综合器在 `_ChipPool` 中分别维护"按 arity 池化的 gate"与"按 primitive 名池化的 block"两个资源池；验证器把两类统一为 `units` 进行拓扑求值，块的多输出按 `outs_list` 顺序写回信号网络。

## H.2 表达式语法增量（对应附录 A.2 的扩展）

A.2 原文规定的是 NAND2 / NAND4 + 头部声明 + 单 LHS 赋值。本项目实际支持以下扩展：

- **字面量 `0` / `1` 注入**：表达式中可写 `NAND4(A, B, C, 1)`，综合时连到 `VCC.0` / `GND.0`；其中 `1` 会进一步参与 VCC 输入替换优化（见 H.4），物理上不直接接电源
- **中间变量赋值**：单 LHS 的赋值若名字不在 `outputs` 中则视为中间变量，例如 `T1 = NAND2(A, A)`；后续表达式可引用 `T1`，但必须先定后用，且名字不可与输入或其它中间变量冲突
- **高层多 I/O 原语**：
  - `DECODE3(C, B, A)`：74138 的 3-8 译码，输出 8 路（按 Y0..Y7 顺序）
  - `MUX4(B, A, C0, C1, C2, C3)`：74153 的 4 选 1，输出 1 路；select = 2·B + A
  - `MUX8(C, B, A, D0, D1, D2, D3, D4, D5, D6, D7)`：74151 的 8 选 1，输出 2 路 `(Y, Ȳ)`；select = 4·C + 2·B + A
- **多 LHS 解构赋值**：
  - 用法：`Y0, Y1, Y2, Y3, Y4, Y5, Y6, Y7 = DECODE3(C, B, A)`、`Y, YBAR = MUX8(...)`
  - 校验：LHS 名字数必须等于该原语的固有输出数（DECODE3 = 8、MUX4 = 1、MUX8 = 2），否则解析阶段报错
  - 单输出原语（MUX4）也允许多 LHS 形式（仅 1 个名字），但通常用单 LHS：`M = MUX4(...)`
  - 多输出原语（MUX8 / DECODE3）必须用对应数量的 LHS 接收；不在 `outputs:` 里声明的 LHS 名字会作为内部信号保留，对应的芯片输出引脚物理上保持悬空
- **嵌套限制**：多输出原语**不可作为 NAND 的子表达式**嵌套，否则综合阶段抛 `SynthError`；如需组合，请用中间变量先接收再使用
- **多行表达式**：括号未闭合的赋值会自动续行直到平衡，便于把长表达式按层次分行书写

## H.3 默认使能（对附录 B.3 的扩展）

附录 B.3 已要求"VCC / GND / NC 引脚显式记录"。本项目对中规模器件再追加一条规则：**用户不显式连使能时，综合器自动写入"芯片使能引脚 → VCC.0 / GND.0"的连线**，规则在 chip 文件的 `Block.default_enables` 字段声明。已实现的默认使能：

- 74138：`G1(pin 6) → VCC.0`，`G2A̅(pin 4) → GND.0`，`G2B̅(pin 5) → GND.0`（默认始终使能）
- 74153：`1G̅(pin 1) → GND.0`（通道 1 默认使能）；本期 74153 一片只暴露通道 1，通道 2 引脚 9/10/11/12/13/15 标记为 NC 占位
- 74151：`Ē(pin 7) → GND.0`（默认始终使能）；双互补输出 Y(pin 5) / Ȳ(pin 6) 按 LHS 顺序绑定，未在 `outputs:` 中声明的那一路对应的芯片引脚保持悬空

显式使能版语法（`DECODE3_EN` / `MUX4_EN` / `MUX8_EN`）作为预留接口暂未实现，留待后续工单。

## H.4 综合 / 验证内部增量

以下均是物理面包板友好的优化，对 verifier 透明，不影响真值表：

- **菊花链扇出**：同一驱动者驱动多个消费者时，连线排成 `src → d0 → d1 → ...` 链而非星型；电气上仍属同一并查集等价类
- **VCC 输入替换**：NAND 门里"应为 1 的输入"自动短接到该门内某个真实信号端点（基于 `NAND(X, ..., X) = NAND(X, ...)` 的代数恒等），物理上避免接 VCC 受电源走线干扰
- **自动 CSE**：综合器以 AST 节点为键缓存已生成的门，结构相等的子表达式只生成一个门
- **统一 units 求值**：`gate`（单输出）通过包装为返回单元素列表的函数后，与 `block`（多输出）共享同一拓扑排序与求值循环

## H.5 器件库扩展流程（对附录 C.3 的扩展）

附录 C.3 原文规定的扩展流程是"加 ChipSpec.gates → 在注册表登记"。引入 Block 后，扩展流程变为：

1. 在 `chips/chip_xxxx.py` 中按需声明 `gates`（小规模门）或 `blocks`（中规模器件块）；两者可同时存在
2. 若新增了高层原语，同步在 `common/parser.py` 的 `_PRIMITIVE_OUTPUT_COUNT` 字典中登记 `"原语名": 输出数`
3. 在 `chips/registry.py` 的 `_MODULES` 列表追加新模块
4. 综合器、验证器、`get_spec` / `list_models` 签名均不变，无需修改

当前已注册器件清单：

- `7400`：4 × NAND2
- `7420`：2 × NAND4
- `74138`：1 × DECODE3（3-8 译码器）
- `74153`：1 × MUX4（双 4 选 1 数据选择器，通道 1）
- `74151`：1 × MUX8（8 选 1 数据选择器，双互补输出 Y / Ȳ）
